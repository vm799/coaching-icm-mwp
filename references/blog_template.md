## Problem Section (200-300 words)
**Subheading:** [What problem are we solving?]

Convert script's problem/hook into prose. Avoid:
- Second person ("You have this problem") — instead: "Most teams struggle with..."
- Exclamation marks — instead: period or em-dash

Write as: "The problem is X. It manifests as Y. The cost is Z."

Include **one stat** if available (cite source).

**Example:**
> Most teams building AI systems struggle with context management. They dump all available data into prompts, hoping the model will figure out what matters. The result: slow responses, bloated contexts, and thousands in wasted API costs. Token efficiency isn't a nice-to-have — it's the difference between a scalable system and a bottleneck.

---

## Solution Intro (150-200 words)
**Subheading:** [What's the solution?]

Introduce concept without jargon (or define jargon).

**From script:** "X is like Y because..." → Convert to prose.

**Example:**
> Context efficiency works the same way a library works. Instead of handing someone every book in the building, you give them the catalog, let them request what they need, and deliver the right book at the right time. In AI systems, this means:
> - Knowing what information matters for this decision
> - Loading only that information
> - Letting the model focus on reasoning, not sifting

## Writing Style (Blog Voice, Not Script Voice)

### Use This
- **Active voice:** "We apply context limits" (not "Context limits are applied")
- **Short paragraphs:** 3-4 sentences, then white space
- **Subheadings:** Every 300 words, guide reader
- **Numbers:** Cite sources ("40% reduction in latency according to [study]")
- **Links:** Hyperlink source docs, previous articles, next article

### Avoid This
- **"We"**: Use "most teams", "the system", or specific case
- **Hyperbole:** "This is the ONLY way" — instead, "This is one proven approach"
- **Jargon without definition:** If new term, define it inline or in parentheses
- **Long sentences:** Break into short ones. Reader scans, doesn't read linearly
- **Filler:** "actually", "basically", "essentially" — delete

### Tone (Varies by Stream)
- **Teaching:** Curious, helpful, non-judgmental. "Let's explore..." not "You must..."
- **Enterprise:** Clear, data-grounded, action-oriented. "ROI is X" not "This is great"

## SEO Basics (Optional, But Helpful)

- **Title:** Include primary keyword (e.g., "Token Efficiency in ICM: How to Scale AI Costs")
- **Headings:** Use H2/H3 structure (not all H1)
- **Meta description:** 155-160 chars, include keyword, describe what reader will learn
- **Internal links:** Link to previous articles (reader journey)
- **Outbound links:** 3-5 to credible sources (improves SEO, adds value)

---

## Checklist Before Publishing

- [ ] Title is <60 chars and includes keyword
- [ ] Intro explains what reader will learn in 2 sentences
- [ ] Sections use subheadings every 300 words
- [ ] Each section has one main idea (not multiple ideas)
- [ ] All sources cited (links included)
- [ ] No script-isms remain ("Hey there", "[PAUSE: ...]", "[SLIDE: ...]")
- [ ] Read aloud: does prose flow naturally?
- [ ] Tone is consistent (not switching between casual and formal)
- [ ] CTA is clear (what should reader do next?)
- [ ] Tags/category set for blog platform

---

## Pro Tips

- **Headlines:** Make them specific. "Context Efficiency" → "Why Selective Context Is Cheaper Than Comprehensive Context"
- **Opening line:** Your best line. Hook reader in first sentence.
- **Visuals:** Every 500 words, break with image/diagram. Not required, but helpful.
- **Code:** Always show working example (not pseudo-code). Reader should be able to copy-paste.
- **Proof-reading:** Read once for flow, once for typos. Blog is published; review carefully.
