## Scene Breakdown

**Script text:** [First 30 seconds of script, exact quote]

### Scene 2: Problem Introduction
**Script text:** [Full problem section, exact quote from script]

**Timing:** 0:15–1:30 (75 seconds)

**Visual Description:**
- What is the visual metaphor for "problem"?
- [E.g., "A person drowning in documents" OR "A system bottleneck visual" OR "A chart showing cost"]

**Elements:**
- Icon or character showing problem state
- Text label or caption: "[Problem statement]"
- Supporting visuals: [charts, data, screenshots]

**Motion:**
- Scene starts with static image (0:15–0:20)
- Problem icon scales or rotates in (0:20–0:30)
- Data/charts appear one by one (0:30–1:00)
- Text caption appears (0:45–0:50)
- Hold state (1:00–1:25)
- Transition to solution (1:25–1:30)

**Notes:** [Keep visuals simple. Avoid clutter. Problem should be immediately recognizable.]

---

### Scene 3: Concept Analogy
**Script text:** [Exact analogy from script, e.g., "X is like Y because..."]

**Timing:** 1:30–3:00 (90 seconds)

**Visual Description:**
- **Left side:** Familiar concept (the analogy, e.g., "library")
- **Right side:** Technical concept (what we're teaching, e.g., "context management")
- **Connection:** Arrow or visual link showing how they're the same

**Elements:**
- Library icon/illustration (left)
- Context system diagram (right)
- Connecting arrow with "because" label
- Text annotations

**Motion:**
- Library scene appears (1:30–1:45)
- User/action in library (1:45–2:15) — e.g., person searching catalog
- Transition to tech side (2:15–2:30)
- Tech diagram appears (2:30–2:50)
- Overlay shows comparison (2:50–2:55)
- Hold (2:55–3:00)

**Narration overlay:** (Script plays over this scene — timestamp match key moments)

**Notes:** [This is teaching moment. Make analogy CLEAR. Exaggerate the visual similarity.]

### Scene 4: How It Works — Step 1
**Script text:** [First "how" step from script]

**Timing:** 3:00–4:15 (75 seconds)

**Visual Description:**
- Step 1 in a process flow
- What's the input? The action? The output?
- [E.g., for "selective context": Input=raw data → Action=filter → Output=lean context]

**Elements:**
- Process boxes or icons (input, action, output)
- Arrow showing flow
- Numbers or labels for each step

**Motion:**
- Input box appears (3:00–3:15)
- Arrow sweeps in (3:15–3:25)
- Action box appears (3:25–3:40)
- Output box appears (3:40–3:55)
- Repeat flow animation (3:55–4:10)
- Hold (4:10–4:15)

**Notes:** [Process animations should feel logical, not too fast. Give viewer time to follow.]

---

### Scene 5: How It Works — Step 2
**Script text:** [Second "how" step from script]

**Timing:** 4:15–5:30 (75 seconds)

**Visual Description:**
- Step 2, building on Step 1
- Visual complexity increases slightly
- [E.g., "Apply rules to prioritize": Show rule icons, prioritization logic]

**Elements:**
- From previous scene (input, action, output) — keep visible
- New layer: rules or logic
- Color code: important vs. less important

**Motion:**
- Previous scene fades to 30% opacity (4:15–4:20)
- Rules layer appears (4:20–4:35)
- Animation shows rule application (4:35–5:00)
- Output highlights in bright color (5:00–5:15)
- Hold (5:15–5:30)

**Notes:** [Show layering. Concept builds. Don't jump to complexity; progress step-by-step.]

### Scene 6: Real-World Example (Code or System)
**Script text:** [Example from script, with [DEMO: ...] marker]

**Timing:** 5:30–7:00 (90 seconds)

**Visual Description:**
- Code snippet OR system screenshot OR workflow diagram
- Highlight the relevant parts
- Show before/after or step-by-step execution

**If code:**
```python
# Example code from research output
# Lines 1-5: input
# Lines 6-10: processing
# Lines 11-12: output
```

**Elements:**
- Code syntax highlighted (colors per language convention)
- Comments explaining each section
- Arrow showing execution flow

**If system:**
- Screenshot of real system UI
- Highlight boxes around relevant buttons/sections
- Annotation explaining what's happening

**Motion:**
- Code/screenshot appears (5:30–5:45)
- Line 1 highlights (5:45–5:55)
- Highlight moves through code (5:55–6:30) — one line every 3-5 sec
- Output highlighted in bold (6:30–6:50)
- Transition to next concept (6:50–7:00)

**Narration overlay:** Script reads through this. Animation follows script timing.

**Notes:** [Code should be readable. Use large font. Don't show too many lines (10 max).]

---

### Scene 7: Impact / Why It Matters
**Script text:** [Impact section: cost, time saved, risk reduced, etc.]

**Timing:** 7:00–8:15 (75 seconds)

**Visual Description:**
- Before/after comparison
- Metric visualization (graph, chart, or simple comparison)
- [E.g., "Cost reduction from $1000/month to $600/month"]

**Elements:**
- Before state: expensive/slow/risky (red/dark colors)
- After state: efficient/fast/safe (green/bright colors)
- % improvement label: "[X]% improvement" or "$[Y] saved"

**Motion:**
- Before state appears (7:00–7:20)
- Transition animation (7:20–7:35) — money flying away, timer slowing, risk meter lowering
- After state appears (7:35–7:55)
- Metric appears with tick mark (7:55–8:10)
- Hold (8:10–8:15)

**Notes:** [Impact visuals should feel satisfying. Reward viewer for paying attention.]

---

### Scene 8: Conclusion / What's Next
**Script text:** [Closing section: recap, call-to-action]

**Timing:** 8:15–8:45 (30 seconds)

**Visual Description:**
- Recap of key points (visual summary)
- Call-to-action (what to do next)
- Next video/resource teaser (if series)

**Elements:**
- Icons for each key takeaway (3 icons)
- Text: "3 Things to Remember"
- CTA button or text: "Start with [action]"

**Motion:**
- Key points icons appear (8:15–8:30) — one every 5 sec
- CTA text appears (8:30–8:40)
- Fade to next scene or end screen (8:40–8:45)

**Notes:** [Keep it brief. Viewer may skip, so make this punchy.]

## Quality Checklist

- [ ] Timing matches script narration (no gaps or overlap)
- [ ] Each scene has clear visual metaphor (not abstract)
- [ ] Motion is smooth (no jarring cuts except intentional transitions)
- [ ] Colors are consistent (style guide followed)
- [ ] Text is readable (size, contrast checked)
- [ ] Character/icon expressions match tone (not contradictory)
- [ ] Transitions are smooth (not abrupt)
- [ ] Final scene leaves clear next-step impression

---

## Animator Notes

This spec is detailed so you understand context, but feel free to:
- **Improve visuals:** If you have a better analogy idea, run it by content team
- **Optimize timing:** If pacing feels off after animating, adjust (must stay <5% off script timing)
- **Add polish:** Details like shadows, subtle motion, are artist's call
- **Simplify:** If spec calls for X but Y is simpler/clearer, go with Y

Keep the **core message** clear. That matters more than perfect execution.
